# 인우님 Quiket v1.0 명세서 패턴 라이브러리

> 이 문서는 인우님이 실제 작성한 `F1_Quiket_기능명세서.xlsx` (v1.0, 2026-05 기준)을 분석해 추출한 패턴이다.
> 신규 명세 작성 시 이 패턴을 그대로 따른다.

## 1. 시트 구성 (실제 인우님 패턴)

```
Sheet 1: 기능명세_목록 v.1.0       (61행 / 7컬럼)
Sheet 2: 기능명세_상세 v.1.0       (1088행 / 11컬럼)
Sheet 3: DB구조                  (도메인 데이터셋 자리)
Sheet 4: 자격증 종류              (도메인 데이터셋 자리)
Sheet 5: 기능명세 질의            (996행 / 7컬럼) ★ v2 신규 시트

도메인 데이터셋 시트(DB구조, 자격증 종류 등)는 **빈 자리만 잡아두고 채우는 건 별도 작업**.

## 4. 항목별 작성 패턴 (실제 사례)

### 패턴 1: 단순 분기형 (스플래시) — 입출력 생략

```
SPLASH        | 앱 시작 | 스플래시 | 설명           | 앱 실행 시 2초간 로고 표시 후 분기 처리
(빈)           |        |         | #1 최초 설치    | 온보딩(#ONBOARD-01)으로 이동
(빈)           |        |         | #2 토큰 유효    | #HOME-MAIN으로 이동
(빈)           |        |         | #3 토큰 만료    | #AUTH-LOGIN으로 이동
```

### 패턴 2: 화면별 카피 분리 (온보딩)

```
ONBOARD       | 앱 시작 | 온보딩  | 설명           | 3장의 온보딩 화면으로 서비스 핵심 가치 소개
(빈)          |        |         | ONBOARD-01    | [main] 내 강의 노트로 만드는 나만의 시험 / [sub] ...
(빈)          |        |         | ONBOARD-02    | [main] 풀수록 업그레이드 되는 ${다람쥐명}의 공부방 / ...
(빈)          |        |         | ONBOARD-03    | [main] 과목별로 쌓아가는 나만의 강의 아카이브 / ...
(빈)          |        |         | 이후 진입화면   | - 마지막 온보딩(#ONBOARD-03) → #AUTH-LOGIN
```

→ `[main]`, `[sub]` 같은 메인/서브 카피 구분자 활용.

(빈)          |     |              | 예외 | - 이메일 형식 오류: '올바른 이메일 형식이 아닙니다' / - 중복 이메일 → '이미 사용 중인 이메일입니다' / - 인증코드 불일치: '인증코드가 일치하지 않습니다' / - 인증코드 유효시간 초과: '인증 시간이 만료되었어요. 재발송해주세요'

### 패턴 4: 플로우형 (카카오/구글 로그인) — 항목 순서 변형

```
AUTH-002      | 인증 | 소셜 가입/로그인 | 설명   | 카카오·구글 소셜 계정 연동 가입/로그인
(빈)          |     |                  | 플로우 | 카카오/구글 계정으로 가입/로그인하기 → 최초 로그인 여부 확인 → 최초 로그인 시 닉네임 설정 화면 → 홈
(빈)          |     |                  | 정책   | - 동일 이메일로 자체 가입 계정이 있는 경우 연동 여부 확인 / - 최초 로그인 시 닉네임 기본값으로 카카오 닉네임 자동 입력 (수정 가능)
(빈)          |     |                  | 입력   | - 카카오 계정 인증 정보 (카카오 인증 SDK 처리, 사용자 입력 없음) / - 최초 로그인 시: 닉네임 (2~12자, 한글/영문)
(빈)          |     |                  | 출력   | - 액세스 토큰/리프레시 토큰 / - 사용자 기본 정보(닉네임, 도토리 잔고 등)
(빈)          |     |                  | 예외   | - 중간에 취소 → 로그인 화면 유지 (오류메시지 미표출) / - 카카오·구글 로그인 실패 -> 오류메시지 표출 (토스트)

→ 표준 순서가 아닌 **설명 → 플로우 → 정책 → 입력 → 출력 → 예외** 순도 허용.

### 패턴 5: 다단계 입력 (과목 추가)

```
SUBJECT-001   | 과목 관리 | 과목 추가 | 설명               | (1) 과목명 입력/학습목적 선택 → (2) 학습 목적 선택 → (3) 시험유형/분야 선택 (3단계 플로우)
(빈)          |          |          | 입력 (Step1)        | - 과목명 입력 필수 (1~30자) / - 학습목적 / 3가지 중 선택 / A. 시험·자격증 대비 / B. 분야 학습 / C. 텍스트 정리·기록
(빈)          |          |          | 입력 (A-Step2)      | 시험유형 선택: '대학교', '중·고등', '자격증', '공무원', '어학', '기타시험'
(빈)          |          |          | 입력 (A-Step3) /    | - 전공 계열 선택: '인문/사회', '자연/과학', ... / - 과목 유형 선택: 직접 입력
(빈)          |          |          | 입력 (A-Step3) /    | - 교육 과정(학년): '중1', '중2', ... / - 과목 유형: 국어, 수학, ...
(빈)          |          |          | 입력 (A-Step3) /    | - 자격증 목록 + 직접 추가
(빈)          |          |          | 입력 (B-Step2,3)    | Step2 - 분야 선택: '인문', '한국사', ... / Step3 - ...
(빈)          |          |          | 입력 (C-Step2,3)    | Step2 - 이용 목적 선택: '업무·실무 활용', ...
(빈)          |          |          | 출력                | 생성된 과목 ID (내부 저장, 화면 미표출) / 과목 기본 정보
(빈)          |          |          | 정책                | 선택한 목적/시험 유형 및 분야 등의 정보는 AI 퀴즈 생성 프롬프트로 자동 반영
(빈)          |          |          | 예외                | - 과목명 미입력 → '과목명을 입력해주세요' / - 건너뛰기 → 입력 필요 상태로 남기고 ...
```

→ **A/B/C 학습 목적별로 Step2, Step3가 분기**될 때 항목명에 분기를 명시.
→ 같은 항목명(`입력 (A-Step3)`)이 여러 행에 걸쳐 등장 가능.

### 패턴 10: 미포함 기능 빈 행 처리

```
SUBJECT-002   | 과목 관리 | 목적/시험유형별 AI 기준 | (빈) | (빈) | Could | (빈) | 미포함 | (빈) | (빈)
(빈)          |          |                       | (빈) | (빈) |       |      |        |      |
(빈)          |          |                       | (빈) | (빈) |       |      |        |      |
(빈)          |          |                       | (빈) | (빈) |       |      |        |      |
SUBJECT-003   | 과목 관리 | 과목 수정/삭제          | 설명 | 화면 우상단 ' ⋮ ' 탭 → 드롭다운 메뉴 ...
```

→ ID는 채우고, 항목·내용·우선순위·API 등은 빈 행으로 자리만 잡아둠 (3~5행).
→ 인우님 v1.0에서: SUBJECT-002, LECTURE-001-B, LECTURE-004, PART-003~005, QUIZ-GEN-002, QUIZ-PLAY-004~005, WRONG-001~004, MY-002~003, SUMMARY-001~002 모두 이 패턴.

### 패턴 11: 정책 카테고리 헤더

```
정책 | [데이터·채점]
     | - 데이터 모델: 퀴즈 세트는 생성 시점에 문제 본문/선택지/정답/해설을 한 세트로 저장
     | - 채점: 클라이언트가 정답 데이터로 채점 (네트워크 독립)
     |
     | [재시도 정책]
     | - 결과 제출 실패: 로컬 답안 보관 후 자동 재시도 (지수 백오프)
     |
     | [저장 시점]
     | - 결과 제출 API 응답 성공 시점에 서버 저장 완료
```

→ 정책이 길어지면 `[카테고리]` 헤더로 구획 분리. 가독성 + 검색성 향상.

## 5. 메모/이슈 활용 패턴

```
| 출력 | (출력 내용) | Must | API명 | MVP | 기획완료 | 인증방법? 이메일에서 '확인'누를지, 메일로 전송된 코드를 입력하게 할지?
| ... | ...        | Must | ...   | 일부포함 | 기획완료 | 카카오만 적용 (안드로이드)
```

→ 결정 필요 = J컬럼(메모/이슈)에 질문 그대로 기록.
→ 부분 적용 사유도 메모에 명시.

## 6. 인우님이 자주 결정해야 하는 항목

작성 중 다음과 같은 결정 필요 사항이 자주 등장한다. 발견 시 **반드시 인우님께 질문**.

| 결정 항목 | 질문 예시 |
|---|---|
| UX 분기 | "에러 시 토스트로 보여줄지, 모달로 보여줄지?" |
| 수치 기준 | "비밀번호 최대 길이는 몇 자까지?" |
| 시간 기준 | "인증코드 유효시간은 5분/10분/30분 중 어느 쪽?" |
| 권한 정책 | "권한 거부 시 기능 일부 차단할지, 전체 차단할지?" |
| MVP 범위 | "이 기능을 MVP에 넣을지 일부포함으로 할지?" |
| 카피 톤 | "에러 메시지를 친근체 vs 공식체 어느 쪽으로?" |
| 저장 위치 | "이 데이터는 로컬에만 저장할지, 서버에도 저장할지?" |
| 채점 시점 | "클라이언트에서 채점할지, 서버에서만 할지?" |
| 적립 룰 | "다시풀기에서 정답률 100%면 도토리 어떻게 지급?" |

## 9. 카피 톤 실제 사례

### 친근체 (해요체) — 인우님이 더 자주 쓰는 톤
```
"지원하지 않는 파일 형식이에요"
"100자 이상 입력해야 퀴즈를 만들 수 있어요"
"기록을 불러올 수 없어요"
"파트를 삭제하면 강의 내용이 삭제되며 복구할 수 없어요. 삭제할까요?"
"인증 시간이 만료됐어요. 재발송해주세요"
"네트워크 연결을 확인해주세요"
"잠시 후 반영돼요"
"이미지를 인식하지 못했어요"
"제출하기 전에 확인해주세요"
"모든 문제를 풀었어요! 답안을 제출할까요?"
```

### 공식체 (다나까체) — 검증/제약 알림에 주로
```
"파일 크기는 50MB 이하만 업로드 가능합니다"
"이미 사용 중인 이메일입니다"
"올바른 이메일 형식이 아닙니다"
"비밀번호가 일치하지 않습니다"
```

→ **기본은 친근체**, 입력 검증·제약 통지 등은 공식체.
