# 상세 기능명세서 작성 스킬 v2 (5단계)

> **v2 업데이트 (2026-05-07 KST 반영)**
> 인우님의 실제 Quiket v1.0 명세서를 레퍼런스로 패턴 라이브러리 전면 보강.
> 신규: 기능명세 질의 시트, 다단계 입력 분기, 케이스 조합 출력, 정책 카테고리 표기, 도메인 정책 ID(POLICY-*), 친근체 카피 톤, 변수 표기 컨벤션.

## 트리거 조건

- "상세 기능명세서 작성", "기능명세 작성"
- "F기능 추가 명세", "스펙 문서 만들어줘"
- "{기능 ID} 명세 수정/추가"
- 마스터 워크플로우의 5단계로 호출

## 핵심 원칙

### 디자이너·개발자가 즉시 작업 가능한 수준의 디테일
**엣지케이스, 입/출력, 정책, 카피에 집중**한다. 두루뭉술한 설명("기능 잘 동작") 금지. 구체적인 조건, 수치, 메시지 카피, 화면 ID 참조까지 명세화.

### 세 시트 구조 (v2: 신규 시트 추가)
1. **기능명세_목록 v.X.X** — 한 줄 요약 인덱스
2. **기능명세_상세 v.X.X** — 항목별 상세 명세
3. **기능명세 질의** — 개발자·디자이너 ↔ 기획자 Q&A 채널 (NEW)

선택 시트 (도메인 데이터셋 또는 정책 부록):
- DB구조 / 자격증 종류 / 변경 이력 등

### 미포함 기능도 자리는 잡아둔다
"미포함" 또는 "보류" 기능이라도 **목록 시트에는 반드시 등록**하고, 상세 시트에는 ID 행 + 빈 행 3~5개를 두어 추후 명세 작성 시 자리를 비워둔다. 이는 인우님의 일관된 작성 패턴이다.

### 기획자 결정 필요 사항은 반드시 질문
명세 작성 중 의사결정이 필요한 항목(특정 정책, 수치 기준, UX 분기 등)이 발견되면 임의로 결정하지 말고 인우님께 명시적으로 질문. 답변 받기 전까지 해당 셀은 "TBD" 또는 메모 컬럼에 질문을 적어둔다.

## 시트 1: 기능명세_목록 v.X.X

### 컬럼 구조

| 컬럼 | 내용 | 예시 |
|---|---|---|
| ID | 기능 식별자 (도메인-번호) | AUTH-001, QUIZ-OPT-001 |
| 섹션 | 기능 분류 | 인증, 홈, 과목 관리, 퀴즈 설정 |
| 기능명 | 기능명 | 자체 회원가입 |
| 한줄 설명 | 30~80자 요약 | 이메일·비밀번호·닉네임으로 계정 생성 (이메일 인증 포함) |
| 포함여부 | MVP / 일부포함 / 보류 / 미포함 | MVP |
| 우선순위 | Must / Should / Could / Won't | Must |
| 관련 US | PRD 유저 스토리 ID | US-001, US-013 |

### 작성 원칙

- ID는 PRD 기능 ID를 그대로 사용
- 섹션은 IA 메뉴/도메인 분류 기준 (예: 앱 시작 / 인증 / 홈 / 과목 관리 / 강의 관리 / 파트 분류 / 퀴즈 설정 / 퀴즈 생성 / 퀴즈 풀기 / 결과 / 오답노트 / 퀴즈 기록 / 게임화 / 마이페이지 / 요점정리)
- 한줄 설명에는 핵심 동작 + MVP 범위 단서 포함 ("...mvp 보류", "...에 한함")
- 포함여부와 우선순위는 같지 않을 수 있음 (예: 미포함이지만 Must — 차후 도입 필수)

## 시트 2: 기능명세_상세 v.X.X

### 컬럼 구조

| 컬럼 | 내용 |
|---|---|
| 기능ID (#화면ID) | 기능 ID + 관련 화면 ID들 (개행으로 나열, 첫 행에만 기재) |
| 섹션 | 기능 분류 (첫 행에만 기재) |
| 기능명 | 기능명 (첫 행에만 기재) |
| 항목 | 설명 / 입력 / 출력 / 정책 / 예외 / 플로우 / #케이스 분기 / 입력 (Step1) 등 |
| 내용 | 항목별 상세 내용 (개행으로 다중 항목 나열) |
| 우선순위 | Must / Should / Could / Won't |
| API 연동 | 필요한 API명 (한 기능에 여러 API 호출 시 모두 나열) |
| 포함여부 | MVP / 일부포함 / 보류 / 미포함 |
| 상태 | 기획중 / 기획완료 / 디자인중 / 디자인완료 / 개발중 / 개발완료 |
| 메모/이슈 | 결정 필요 사항(TBD), 미해결 이슈, 참고 서비스, 부분 적용 사유 |

### 항목 명명 컨벤션 (v2 강화)

#### 기본 항목
| 항목명 | 용도 |
|---|---|
| 설명 | 기능 목적 + 핵심 동작 + 진입점 |
| 입력 | 클라이언트 → 서버 / 사용자 입력 / 자동 입력 |
| 출력 | 서버 → 클라이언트 / 화면 전환 / UI 피드백 |
| 정책 | 규칙·기본값·제약·시간·횟수·권한 등 |
| 예외 | 엣지케이스, 에러 케이스, 메시지 카피 포함 |
| 플로우 | 다단계 화면 전환 (1)~(N) |

#### 분기·다단계 항목 (v2 신규)
| 항목명 패턴 | 용도 | 예시 |
|---|---|---|
| `#1 케이스명` | 단순 분기 (스플래시 분기 등) | `#1 최초 설치`, `#2 토큰 유효`, `#3 토큰 만료` |
| `#옵션명` | 옵션·모드 분기 | `#객관식`, `#O/X 퀴즈`, `#한 문제씩`, `#한번에` |
| `#모드명 모드` | 풀이 모드별 인터랙션 | `#한문제씩 모드`, `#한번에 모드` |
| `#플로우명 플로우` | 플로우별 단계 | `#AI에게 맡기기 플로우`, `#직접 분류 플로우` |
| `입력 (Step1)` | 다단계 입력 | `입력 (Step1)`, `입력 (A-Step2)`, `입력 (A-Step3)`, `입력 (B-Step2,3)`, `입력 (C-Step2,3)` |
| `출력 (#조건)` | 케이스 조합 출력 | `출력 (#안푼0개/마킹0개)`, `출력 (#안푼≥1개/마킹0개)`, `출력 (#안푼0개/마킹≥1개)`, `출력 (#안푼≥1개/마킹≥1개)` |
| `화면ID` | 화면별 카피 분리 (온보딩 등) | `ONBOARD-01`, `ONBOARD-02`, `ONBOARD-03`, `이후 진입화면` |
| `유형` | 옵션 단일 분기 | `유형` (쪽지시험 단답형/빈칸채우기) |

### 항목별 작성 가이드 (v2 강화)

#### 설명 (Description)

- 기능의 목적과 핵심 동작을 2~5줄로 정리
- **진입점은 반드시 명시** (어디서 이 기능에 접근하는지)
- **다중 진입점은 `진입점 A:`, `진입점 B:` 처럼 분리**
- **[구성]·[ 케이스]·[차별 인터랙션] 같은 카테고리 헤더로 가독성 향상**

예시:
```
앱의 메인 진입점.

[구성]
- 앱바: 앱 로고, 알림 아이콘
- 히어로 섹션+퀵액션 버튼: 강의 업로드(#LECTURE-UPLOAD), ...
- 최근 활동 탭: 진행중 퀴즈, 완료 퀴즈
```

```
퀴즈 출제 범위(챕터/파트) 선택
- 진입점 A: 홈 → 과목 선택 → #QUIZ-SCOPE-A
- 진입점 B: 과목 상세 → 챕터 선택 → #QUIZ-SCOPE-B
```

#### 입력 (Input) — 클라이언트 → 서버

다음 중 해당하는 모든 입력을 작성.

| 유형 | 작성 예시 |
|---|---|
| 사용자 입력 | 이메일, 비밀번호, 닉네임 / 답안 선택, 다음/이전 이동, 중도 종료 |
| 입력 검증 규칙 | 이메일: 형식 검증 + 중복 불가 / 비밀번호: 8자 이상 64자 이하, 영문+숫자 필수 |
| 자동 입력 | 디바이스 ID, FCM 토큰, 사용자 ID(토큰), 풀이 세션 ID |
| 시스템 컨텍스트 | 현재 위치, 부모 세트 ID, 새 풀이세션 ID 등 |

다단계 플로우는 `입력 (Step1)`, `입력 (A-Step2)` 처럼 단계 분리.

#### 출력 (Output) — 서버 → 클라이언트 또는 UI 반응

| 유형 | 작성 예시 |
|---|---|
| API 응답 데이터 | 액세스 토큰, 사용자 기본 정보, 진행률 |
| 화면 전환 | 가입 완료 → #HOME-MAIN |
| UI 피드백 | 토스트("저장되었어요"), 모달, 칩 상태 변경, 진행 바 |
| 비동기 알림 | 푸시 알림, 인앱 토스트 |

케이스 조합은 `출력 (#안푼0개/마킹0개)` 처럼 조건을 항목명에 명시.

#### 정책 (Policy)

기능이 따라야 할 규칙. **카테고리 헤더 `[카테고리]` 사용 권장 (v2)**.

| 카테고리 (`[]` 표기) | 작성 예시 |
|---|---|
| `[기본값]` | 풀기 방식 기본값: 한 문제씩 |
| `[제약 조건]` | 최대 100문제, 최소 1문제 |
| `[시간/유효기간]` | 인증코드 유효시간 10분 |
| `[횟수 제한]` | 비밀번호 5회 실패 시 계정 잠금 |
| `[처리 순서]` | 인증 완료 전까지 다음 단계 진입 불가 |
| `[재시도 정책]` | 재발송 시 이전 코드 무효 처리 |
| `[데이터·채점]` | 정답·해설은 사전 저장 값 사용, 풀이 시 AI 재호출 없음 |
| `[로컬 저장 대상]` / `[서버 저장 대상]` | 풀이 중 답안: 로컬 / 결과 제출 후: 서버 |
| `[지급 룰 - 서버 재채점 결과 기준]` | 첫 풀이 정답 1문항당 +1도토리 |
| `[저장 시점]` | 결과 제출 API 응답 성공 시점에 서버 저장 완료 |
| `[차별 인터랙션]` | 답 선택 즉시 정답/오답 시각적 피드백 |
| `[권한]` | 알림 권한 미허용 시 푸시 비활성 |

#### 예외 (Edge Cases)

다음 분기를 누락 없이 검토.

| 예외 카테고리 | 검토 사항 |
|---|---|
| 입력값 오류 | 형식 오류, 빈값, 범위 초과, 특수문자 |
| 중복/충돌 | 이미 존재하는 ID, 동시성 문제 |
| 권한/인증 | 미인증, 권한 부족, 토큰 만료 |
| 네트워크 | 끊김, 타임아웃, 서버 오류 |
| 데이터 부재 | Empty State, 0건 응답 |
| 데이터 손상 | 해설 누락, 로컬 DB 손상, AI 응답 스키마 불일치 |
| 한도 초과 | 일일 횟수, 월간 한도, 파일 크기 |
| 외부 의존 실패 | OAuth 실패, 결제 실패, AI 응답 실패, OCR 실패 |
| 사용자 이탈 | 화면 이탈, 백그라운드 진입, 앱 종료 |

각 예외는 다음 형식으로 작성 (인우님 카피 톤 = **친근체 우선**):

```
- {조건}: '{사용자에게 보이는 메시지}'
- {조건}: '{메시지}' + {복구 동작}
```

예시:
```
- 50mb 초과: "파일 크기는 50MB 이하만 업로드 가능합니다"
- PDF 외 형식: "지원하지 않는 파일 형식이에요"
- 중복 이메일 → '이미 사용 중인 이메일입니다'
- 인증코드 유효시간 초과: '인증 시간이 만료됐어요. 재발송해주세요'
```

#### 플로우 (Flow) — 다단계 기능에 한해

- 단계별 화면 전환을 (1)~(N)으로 정리
- 각 단계의 입력/출력 명시
- 분기는 별도 행 `#분기명 플로우` 항목으로 분리

예시:
```
A방법
(1) 과목 선택(#LECTURE-SUBJECT) → (2) 챕터명 입력 + 업로드 방식 선택(파일/이미지/텍스트) → (3) 파트 분류
B방법
(1) 과목 상세 진입 → (2) 챕터 추가 → (3) 업로드 방식 선택 → ...
```

#### API 연동

- 필요한 API명을 명시 (예: `로그인 API`, `이메일 중복확인 API`, `퀴즈 생성 API`)
- 한 기능에 여러 API 호출 시 모두 나열

#### 메모/이슈

- 결정 필요 사항: "**TBD**: 인증방법? 메일에서 '확인' 누를지, 코드 입력할지?"
- 디자인 이슈: "디자인 시안 도착 후 카피 확정"
- 부분 적용 사유: "카카오만 적용 (안드로이드)"
- 참고 서비스: "참고 서비스: ${서비스명}의 가입 플로우"

### v2 추가: 카피 톤 컨벤션

인우님의 명세서는 **친근체와 공식체를 혼용**하되, **사용자에게 직접 보이는 토스트·모달·다이얼로그는 친근체 우선**.

| 톤 | 사용처 | 예시 |
|---|---|---|
| 친근체 (해요체·아요체) | 토스트, 빈 상태, 설명, 권유 | "지원하지 않는 파일 형식이에요", "100자 이상 입력해야 퀴즈를 만들 수 있어요", "기록을 불러올 수 없어요", "삭제할까요?" |
| 공식체 (다나까체·합니다체) | 검증 메시지, 정책 안내 | "파일 크기는 50MB 이하만 업로드 가능합니다", "이미 사용 중인 이메일입니다" |
| 혼용 OK | 동일 기능 내에도 혼합 사용 가능 | (인우님 v1.0에서도 두 톤이 함께 등장) |

### v2 추가: 변수·플레이스홀더 표기

명세서에서 동적 값을 표시할 때 `${변수명}` 사용.

```
"선택된 과목 > ${과목명}"
"풀수록 업그레이드되는 ${다람쥐명}의 공부방"
"+'${입력값}' 직접 추가하기"
"기타 시험: ${직접입력}"
```

### v2 추가: 도메인 공통 정책 ID (POLICY-*)

기능별 명세에 반복 등장하는 공통 정책은 **별도 ID로 분리**한다.

```
POLICY-LOCAL — 로컬·서버 저장 분리 정책
POLICY-RETRY — 재시도 정책 (네트워크 실패 등)
POLICY-AUTH  — 토큰 만료 처리 정책
```

다른 기능에서는 "POLICY-LOCAL 참고" 한 줄로 참조.

### v2 추가: 미포함 기능 빈 행 컨벤션

상세 시트에서 미포함/보류 기능은 **ID 행만 한 줄 채우고, 항목·내용은 빈 행 3~5개 두기**. 이는 추후 명세 작성 시 자리를 비워두는 인우님의 일관된 패턴.

```
[행 N]   SUBJECT-002 | 과목 관리 | 목적/시험유형별 AI 기준 | (빈) | (빈) | Could | (빈) | 미포함 | (빈) | (빈)
[행 N+1] (빈 행 — 항목·내용 모두 비움)
[행 N+2] (빈 행)
[행 N+3] (빈 행)
[행 N+4] SUBJECT-003 | ...
```

## 시트 3: 기능명세 질의 (v2 신규)

### 목적

개발자·디자이너가 명세를 보면서 발생하는 질문을 인우님이 한 곳에서 받고 답하는 채널. 카톡·미팅 메모 분산 방지.

### 컬럼 구조

| 컬럼 | 내용 |
|---|---|
| 순번 | 1, 2, 3, ... (자동 부여) |
| 기능ID (#화면ID) | 질문이 향하는 기능 (관련 기능 여러 개면 ` / ` 로 구분) |
| 섹션 | 기능 분류 |
| 기능명 | 기능명 |
| 문의 제목 | 한 줄 요약 (예: "강의-파트 관계", "도토리 적립 시점") |
| 문의사항 | 질문 본문. 본인이 이해한 바를 먼저 적고 "이해한 바가 맞는지 확인 부탁드립니다" 형태가 일반적 |
| 답변 | 인우님(기획자) 답변. 보통 "1:1 관계가 아닙니다" 같은 단언 + 풀이 |

### 작성 원칙

- 빈 시트로 두고 순번 1~30 정도 미리 채워두기 (인우님 v1.0에서는 996까지 채워뒀음 — 50~100 정도가 현실적)
- 새 명세서 생성 시 순번 칸만 1~30 행에 미리 입력해두면 즉시 활용 가능
- 답변이 명세에 영향을 주면 → 해당 기능의 상세 시트에도 반영하고 답변 칸에 "→ 상세 시트 반영 완료" 추가

## 작성 절차

### Step 1: 이전 단계 산출물 활용

- `02_PRD.md`의 기능 목록 → 기능명세_목록 시트 채우기
- `03_IA.md`의 화면 ID → 기능명세_상세 시트의 화면 ID 컬럼에 활용
- `04_유저플로우.md`의 분기 케이스 → 예외/정책 항목에 풀어냄

### Step 2: 시트 1 (목록) 작성

PRD 기능 목록을 그대로 옮기면서 ID, 섹션, 우선순위, 포함여부 정리. **미포함·보류 기능도 빠짐없이 등록**.

### Step 3: 시트 2 (상세) 작성

각 기능별로 다음 순서:
1. 첫 행: 기능 ID(+화면 ID 개행), 섹션, 기능명, 항목="설명", 내용
2. 둘째 행부터: 항목="입력"/"출력"/"정책"/"예외"... 순으로
3. 분기가 있으면 항목="#분기명" 또는 "입력 (Step1)" 형태로 분리
4. **미포함 기능은 ID만 채우고 빈 행 3~5개 두기**

항목 순서는 인우님 v1.0 패턴을 따른다:
- 표준: 설명 → 입력 → 출력 → 정책 → 예외
- 카카오 가입형: 설명 → 플로우 → 정책 → 입력 → 출력 → 예외
- 단순 분기형: 설명 → #1 → #2 → #3 (입출력 생략)

### Step 4: 시트 3 (기능명세 질의) 준비

빈 시트 + 헤더 + 순번 1~30 미리 채우기.

### Step 5: 결정 필요 사항 질문

작성 중 결정 필요 항목 발견 시:
1. 해당 셀은 "TBD" 또는 비워두고
2. 메모/이슈 컬럼에 질문 기록
3. 인우님께 묻고 답변 받은 후 채움

### Step 6: 엑셀 파일 생성

`/mnt/skills/public/xlsx/SKILL.md`을 view 도구로 먼저 읽어 환경 가이드를 따른 후 `scripts/create_spec_xlsx.py` 실행 → 빈 템플릿 생성 → 데이터 채우기.

### Step 7: 검증 체크리스트

- [ ] 모든 Must 등급 기능에 설명/입력/출력/정책/예외가 모두 작성되었는가
- [ ] 화면 ID가 IA에 실제 존재하는가
- [ ] API명이 일관성 있게 작성되었는가 (대소문자, 한/영 혼용 통일)
- [ ] 카피 톤이 친근체/공식체 컨벤션을 따르는가
- [ ] 미포함 기능도 목록에 있고, 상세에 자리(빈 행) 잡혀있는가
- [ ] 케이스 조합 출력의 분기 누락 없는가 (예: 안푼·마킹 4분면)
- [ ] 정책 카테고리 `[ ]` 헤더로 가독성 챙겼는가 (선택이지만 권장)
- [ ] POLICY-* 같은 공통 정책으로 분리할 수 있는 항목이 있는가
- [ ] 기능명세 질의 시트가 헤더+순번 채워진 상태로 비어있는가
